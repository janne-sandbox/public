Code Review System — Overview

Overview

Automated 6 Languages Local-First Requirement-Driven

The Example Code Review Tool is an automated review framework that evaluates codebase changes against project requirements. It supports six targets: TypeScript, Java, Rust, React, C++, and C#. Reviews run locally without internet access by default, while an explicit online mode can use the hosted Deepseek API.

Core purpose: Provide fast, deterministic, requirement-based code reviews that catch common issues in each language before they reach the main branch.

Three review verdicts

VerdictMeaningAction
APPROVEDNo issues found. Code passes all checks.Can merge
APPROVED WITH CHANGESIssues found; suggestions provided. Review and implement suggestions.Request changes
DENIEDCritical issues prevent approval (security, compilation, etc.).Block merge

Quick Start

Review all languages (default)

./run-review.sh

Review C++ offline

LANGUAGE=cpp INTERNET=false ./run-review.sh

Review C# with the online Deepseek API

export DEEPSEEK_API_KEY='your-key' LANGUAGE=csharp INTERNET=true DEEPSEEK=true ./run-review.sh

Results are written to a per-run folder at ./review/results/<timestamp>/. Each run produces one to six report files depending on the selected LANGUAGE value. If another run starts within the same minute, the folder gets a numeric suffix such as 202605261955-1/.

Review with optional Deepseek analysis

# Default: DEEPSEEK=true, auto-uses Ollama if available LANGUAGE=all ./run-review.sh # Force regex-only mode LANGUAGE=all DEEPSEEK=false ./run-review.sh

Six review agents

① TypeScript Agent

  • Scope: TypeScript and TSX files anywhere in the reviewed project
  • Checks: Strict mode, type safety patterns, tests, and requirement coverage
  • Based on: REQ-006, REQ-012, REVIEW-001

② Java Agent

  • Scope: Java source and test files anywhere in the reviewed project
  • Checks: Java 21 patterns, Jackson usage, SQL injection prevention, tests
  • Based on: REQ-033, REQ-058, OWASP-REQ, REVIEW-002

③ Rust Agent

  • Scope: IoT device simulator
  • Checks: Memory safety, error handling, async patterns, and tests
  • Based on: REQ-051, REVIEW-004

④ React Agent

  • Scope: React JavaScript, JSX, TypeScript, and TSX files
  • Checks: Accessibility (WCAG), security, performance, component organization
  • Based on: REQ-011, WCAG-REQ, OWASP-REQ, REVIEW-003

⑤ C++ Agent

  • Scope: C/C++ source and header files
  • Checks: Unsafe functions, raw ownership, casts, tests, and optional Deepseek semantic review
  • Command: LANGUAGE=cpp ./run-review.sh

⑥ C# Agent

  • Scope: C# source and project files
  • Checks: Async correctness, SQL construction, nullable references, tests, and optional Deepseek semantic review
  • Command: LANGUAGE=csharp ./run-review.sh

Requirements-driven reviews

Each agent loads requirements from two sources:

  1. ./requirements/functional/ — optional functional requirement Markdown files
  2. ./requirements/non-functional/ — non-functional, security, quality, and review-policy requirements

Requirements are scanned for checkpoints — code blocks, lists, and instructions — then matched against the codebase. The review verdict is based on how many requirements are met or violated.

Example: The TypeScript agent checks if tsconfig.json has "strict": true (from REQ-006). If not, a major finding is issued with a suggested code fix.

Output format

Review reports are written as Markdown files with the following structure:

./review/results/202605261949/ ├── review-typescript-approved-202605261949.md ├── review-java-approved-202605261949.md ├── review-react-approved-202605261949.md ├── review-rust-approved-202605261949.md ├── review-cpp-approved-with-changes-202605261949.md └── review-csharp-approved-with-changes-202605261949.md

Each report includes:

  • Timestamp, verdict, and analysis mode — when the review ran, overall result, and whether regex-only or hybrid Deepseek analysis was used
  • Summary — issue count by severity (critical, major, minor)
  • Suggested changes — with code snippets and explanations
  • Detailed findings — per-issue description and file location
  • Requirements coverage — which requirements were checked

Parameters and configuration

ParameterValuesDefaultDescription
LANGUAGEtypescript|java|rust|react|cpp|csharp|allallWhich language(s) to review
INTERNETtrue|falsefalseSelect local Ollama (false) or the hosted Deepseek API (true)
DEEPSEEKtrue|falsetrueEnable optional Deepseek analysis; unavailable providers fall back to deterministic checks
DEEPSEEK_OLLAMA_MODELOllama model namedeepseek-coder:1.3b-baseOffline model override
DEEPSEEK_ONLINE_MODELDeepseek API model namedeepseek-v4-flashOnline model override
Data boundary: INTERNET=false keeps analysis local and uses Ollama when available. INTERNET=true sends requirement headings and bounded source samples to the hosted Deepseek API when DEEPSEEK_API_KEY is configured.

SeverityImpactAction
CriticalSecurity or safety issue; code does not compile; hard requirement violatedDENIED — must fix before merge

Integration with CI/CD

Example GitHub Actions workflow:

jobs: review: runs-on: ubuntu-latest steps: - uses: actions/checkout@v3 - name: Run code review run: LANGUAGE=all INTERNET=false ./run-review.sh - name: Upload results uses: actions/upload-artifact@v3 with: name: review-reports path: ./review/results/
Tip: Set the CI job to fail if any report contains DENIED verdict. Parse report files to automate approval workflows.
