AI-assisted software development

Human reviewed Cross-platform Editor integrated Verify every change

AI assistants can explain unfamiliar code, propose focused edits, create tests, and review changes. They do not understand every project constraint and can produce insecure, incorrect, or invented output. Treat generated code as an untrusted contribution that must pass the same review, testing, and security controls as human-written code.

Before using hosted AI: confirm the organization's data-handling policy. Do not submit secrets, credentials, personal data, proprietary source, or customer material unless the selected service and account are approved for that data.

Learning path

1. Prepare VS Code

Install the editor correctly for macOS, Windows, WSL, Ubuntu, or RHEL.

2. Isolate VS Code settings

Use profiles and conservative AI permissions before enabling agent features.

3. Try Codex

Add relevant editor context, request a small task, and inspect the diff.

4. Try Copilot

Compare inline completion, Ask, Plan, and Agent workflows.

5. Try Claude

Use the graphical extension or the optional terminal CLI.

6. Explore local models

Install Ollama, download a suitable model, and verify its local API.

7. Configure Continue

Use local Ollama and hosted Anthropic models from one VS Code assistant.

8. Choose a model and mode

Match reasoning, planning, tools, and specialized roles to the task.

9. Continue across chats

Record verified status and next steps once so another assistant can resume safely.

10. Understand MCP

Connect assistants to external tools and data through explicit trust boundaries.

11. Understand temperature

Control model variation where the selected interface supports sampling settings.

12. Adopt a safe workflow

Scope, prompt, review, test, scan, and document the result.

A useful first task

Start with a reversible, bounded change in a clean Git worktree. Give the assistant the acceptance criteria and exact validation command.

Read the project instructions and the selected files first. Explain the current behavior. Add one focused test for <behavior> without changing public APIs. Run the focused test and report the exact result. Do not change unrelated files.

Inspect git diff, run the project checks yourself, and only then keep or commit the change.
