AI DEVELOPMENT / Models and operating modes

Model, role, and mode are different

Terms such as “thinking,” “planning,” and “agent” describe different layers. A model generates responses. A role assigns that model to a feature. A mode controls which tools and actions the application makes available.

LayerQuestion it answersExample
ProviderWhich service or runtime receives the request?Ollama or Anthropic.
ModelWhich trained model processes the prompt?DeepSeek R1, Qwen3 Coder, or Claude.
CapabilityWhat input and output can the model/provider actually handle?Tool calls, image input, or long context.
RoleWhich IDE feature should use the model?chat, edit, apply, or autocomplete.
ModeWhat is the assistant allowed to do in this interaction?Chat, Plan, or Agent.
Key point: Plan and Agent are normally operating modes, not model names. In Continue, both use the selected chat model, provided the model/provider can use the required tools.

Thinking or reasoning models

A thinking or reasoning model spends additional inference effort before producing its answer. It is useful when a task needs dependency analysis, multi-step debugging, mathematical reasoning, trade-off evaluation, or a careful implementation plan.

  • Use it for ambiguous or high-impact work where a shallow first answer is risky.
  • Expect higher latency and often greater compute or token usage.
  • Do not confuse reasoning with proof: the final answer can still be wrong.
  • Ask for assumptions, evidence, risks, and verification—not hidden internal chain-of-thought.

In the sample Continue configuration, DeepSeek R1 8B is the local reasoning-oriented choice. A larger hosted model can also be selected for difficult planning or agent tasks.

Chat, Plan, and Agent modes

ModeToolsUse it forMain risk
ChatNo autonomous tools in Continue Chat mode.Questions, explanations, brainstorming, and manually supplied context.An answer may sound correct without examining enough repository evidence.
PlanRead-only exploration tools.Tracing behavior, investigating an issue, comparing approaches, and creating an implementation plan.Read access can still expose sensitive files or external data.
AgentRead, write, terminal, and configured MCP tools.Implementing a reviewed plan, running tests, and completing multi-step work.It can modify files, run commands, or change external systems.
Agent mode is an execution loop. The application sends tool definitions to the model, the model requests a tool call, the application asks for permission when required, executes it, and returns the result to the model. The model does not directly bypass the host's permissions.

Specialized Continue roles

RolePurposePreferred characteristic
chatConversation plus the model used by Plan and Agent modes.Good instruction following and tool use.
editGenerate a requested code change.Code quality and precise instruction following.
applyMerge a generated change into the correct file locations.Fast, reliable diff application.
autocompleteProduce inline code suggestions while typing.Very low latency and fill-in-the-middle support.
embedConvert code or documents into vectors for semantic search.Retrieval quality; it does not generate chat responses.
rerankReorder retrieved results by relevance.Accurate relevance scoring; it does not implement code.

A single model may hold several roles, as in the sample configuration, but a specialized and faster model can be better for autocomplete, apply, embeddings, or reranking.

Choose a workflow

TaskSuggested model characteristicSuggested mode
Explain one functionFast code-capable model.Chat.
Understand an unfamiliar serviceStrong context handling and reasoning.Plan.
Design a risky migrationStrong reasoning model.Plan first; human approval before Agent.
Implement a small reviewed fixReliable code and tool-use model.Agent with Ask First permissions.
Generate inline suggestionsSmall, fast code-completion model.Autocomplete role, not Agent mode.
Search a large codebaseEmbedding and optionally reranking models.Context retrieval feeding Chat, Plan, or Agent.
  1. Start in Chat for a narrow question or Plan for repository exploration.
  2. Check that the proposed plan names files, risks, and validation commands.
  3. Switch to Agent only when the scope and permissions are acceptable.
  4. Review commands, diffs, and test evidence independently.

Capabilities must be real

A configuration entry such as tool_use or image_input describes an expected capability; it does not teach the model that capability. Model size alone also does not guarantee reliable planning or tool use.

  • Confirm support in the exact provider and model version.
  • Test tool calls using harmless read-only actions before allowing writes.
  • Evaluate models on representative project tasks rather than rankings alone.
  • Keep a fallback when a local model cannot satisfy the requested mode.
