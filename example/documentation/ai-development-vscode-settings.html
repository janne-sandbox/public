What settings.json controls

VS Code stores editor preferences and extension configuration in settings.json. The graphical Settings editor and the JSON file modify the same values. JSON is useful for review, backup, and exact configuration, but a single incorrect comma or value can prevent VS Code from writing settings.

Choose the correct scope

ScopeWhere it appliesTypical locationUse for
UserEvery window using the default profile.VS Code user-data directory.Personal appearance and conservative defaults.
ProfileWindows using the selected profile..../User/profiles/<profile-id>/settings.jsonLanguage-, customer-, or AI-specific settings and extensions.
RemoteThe active WSL, SSH, or container environment.Remote VS Code server user settings.Remote interpreter paths and remote-only tools.
WorkspaceThe current repository or workspace..vscode/settings.jsonShareable project formatter, lint, and schema settings.
Workspace folderOne folder in a multi-root workspace.That folder's .vscode/settings.jsonFolder-specific overrides.

Later and more specific scopes usually override earlier ones. Language-specific settings have additional precedence rules. Use @modified in the Settings UI to find non-default values.

WSL detail: Windows UI settings and WSL remote settings are different scopes. Put Linux paths such as /usr/bin/python3 in the WSL remote scope, not the Windows user scope.

Why use a separate AI profile?

Create a profile such as AI Agent Lab and associate it only with repositories where agent features are intentionally used.

  • It prevents experimental AI settings and extensions from silently applying to every project.
  • It makes the active extension set, MCP servers, hooks, shortcuts, and permissions easier to audit.
  • It provides a fast escape hatch: switch back to a conservative profile when opening unknown code.
  • It avoids sharing macOS-only paths with Windows, WSL, or Linux profiles.
  1. Open Profiles and choose New Profile.
  2. Start from an Empty Profile or copy only the needed categories.
  3. Install only the AI and language extensions required for that workflow.
  4. Apply conservative settings first, then add narrow approvals after observing repeated safe actions.
  5. Associate the profile with specific trusted folders or workspaces.
A profile is organization, not isolation. Profiles run as the same operating-system user and can reach the same files, credentials, network, and processes. Use Workspace Trust, manual approvals, sandboxing, least-privilege credentials, containers or VMs, and OS controls for real security boundaries.

Why the approval combination is dangerous

SettingEffectAssessment
chat.permissions.default: "autoApprove"Starts new chats in Bypass Approvals.All tool calls can run without confirmation.
chat.tools.global.autoApprove: trueGlobally approves every tool.VS Code describes this as disabling critical protections and does not recommend it.
chat.tools.terminal.autoApprove: { "*": true }Attempts to approve every terminal command.A broad command rule can permit downloads, builds, credential access, or destructive actions.
chat.tools.terminal.enableAutoApproveEnables command-rule auto approval.Use narrow commands rather than a wildcard.
chat.tools.edits.autoApproveApproves all files except listed sensitive patterns.Useful deny patterns, but Bypass Approvals/global approval can override finer controls.
chat.tools.urls.autoApproveApproves matching URL requests and/or returned content.Returned web or localhost content can contain prompt injection.
chat.agent.sandbox.allowAutoApproveAllows auto approval for commands already running inside an enabled sandbox.It does not enable sandboxing; the sample has no chat.agent.sandbox.enabled.
chat.tools.terminal.ignoreDefaultAutoApproveRules: falseKeeps VS Code's built-in terminal approval rules.Helpful, but not a substitute for removing broad bypass settings.
Finer deny rules do not rescue Bypass Approvals. VS Code documents that Bypass Approvals and Autopilot override per-tool, URL, terminal, and sandbox approval settings. A malicious instruction in source code, a fetched page, an MCP result, or a hook can therefore lead directly to actions.

Schema settings and portability

  • yaml.schemas associates YAML patterns with schemas for completion and validation.
  • The sample points to two extension versions for the same files. Those absolute paths break after extension upgrades, on another username, on Intel macOS, Windows, WSL, or Linux, and may cause competing schema matches.
  • Prefer the schema automatically registered by Continue, a stable project schema, or one current extension path in a machine-specific profile.
  • yaml.disableSchemaDetection disables automatic schema detection for CI workflow files. This can remove useful validation exactly where configuration mistakes have deployment or secret-handling consequences.
  • continue.enableTabAutocomplete enables Continue inline completions; continue.enableConsole enables its diagnostic console and can expose prompt or context details during troubleshooting.

Sandboxing is currently available for macOS and Linux, including WSL 2—not native Windows. Even with sandboxing, inspect diffs and use least-privilege credentials.

Review checklist

  1. Fix all JSON errors and unknown-setting warnings before continuing.
  2. Search Settings for @modified and review every non-default value.
  3. Confirm the active profile and remote scope in the current window.
  4. Keep Workspace Trust enabled and do not trust a parent directory containing unrelated repositories.
  5. Remove global and wildcard approvals; add narrow rules only after repeated observation.
  6. Audit hooks, skills, MCP servers, extension publishers, and imported configurations.
  7. Do not store API keys, passwords, tokens, or connection secrets in settings or synchronized profiles.
  8. Export a profile only after checking the export for machine paths and sensitive values.
