AI DEVELOPMENT / Responsible workflow

AI assists; engineering controls decide

A useful AI workflow makes changes easier to inspect and prove. It does not bypass requirements, peer review, threat modeling, accessibility checks, licenses, or the definition of done.

1. Prepare

Use a clean branch, read project instructions, identify sensitive data, and define the acceptance criteria.

2. Constrain

Name the files, behavior, compatibility boundaries, and commands in scope. Ask for a plan when risk is unclear.

3. Inspect

Review commands and each changed line. Reject unrelated refactors, hidden dependencies, or weakened controls.

4. Prove

Run tests, linters, builds, dependency checks, and security scans independently. Record exact results.

Prompt template

Goal: <one observable behavior> Read first: <instructions, requirements, and relevant files> Constraints: - change only <scope> - preserve <API/data/security boundary> - do not add dependencies without approval - never read or print secrets Acceptance: - <testable outcome> - run: <exact validation command> First response: Explain the current behavior and propose a plan. Do not edit yet.

Review checklist

AreaQuestions
ScopeAre all changed files necessary? Did generated formatting hide a behavioral change?
CorrectnessDo tests cover success, failure, boundaries, concurrency, and cleanup where relevant?
SecurityWere authorization, input validation, escaping, cryptography, logging, and secret handling preserved?
DependenciesIs each new dependency necessary, maintained, compatible, and allowed by policy and license?
OperationsAre errors observable? Are migrations, rollback, performance, and resource usage understood?
EvidenceWere validation commands actually run, and are failures or skipped checks stated accurately?

Data and access boundaries

  • Classify the repository before sending source to a hosted assistant.
  • Exclude credentials, environment files, production logs, personal data, and customer content.
  • Use organization-managed accounts and approved retention settings.
  • Grant filesystem, terminal, network, and cloud access only for the current task.
  • Review tool calls as carefully as generated code; a correct-looking result can still come from an unsafe command.
  • For Ollama, keep the API on loopback unless a secured, authenticated deployment has been designed.

Apply the local review suite

AI-assisted changes still need deterministic analysis. From example/, run the review agents and then the language's real build and test commands.

npm test DEEPSEEK=false ./run-review.sh # Optional local semantic analysis after Ollama is verified DEEPSEEK=true INTERNET=false ./run-review.sh

Read every generated report. A tool verdict is evidence for a reviewer, not permission to merge.
