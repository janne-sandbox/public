Development Process - PDCA Delivery Cycle

Delivery Loop

PDCA is applied to each meaningful increment. The cycle is deliberately small enough that findings can return to implementation quickly, while each pass still produces the evidence needed for a release decision.

Plan

Goal: create a shared, testable understanding of the change before implementation begins.

Activities

  1. Collect source material and identify stakeholders affected by the change.
  2. Describe the desired outcome, current constraint, and boundaries of the increment.
  3. Write or refine functional and non-functional requirements.
  4. Record significant architectural choices and known risks.
  5. Define acceptance criteria and the evidence required at review.

Exit criteria

  • The change has an owner and a bounded scope.
  • Requirements are identifiable and testable.
  • Dependencies, risks, and assumptions are visible.
  • The planned tests and review approach are understood.

Do

Goal: implement the smallest complete increment that can be checked against the plan.

Activities

  1. Implement the change using the repository's established conventions.
  2. Add focused unit tests for decisions and edge cases.
  3. Add integration tests at changed boundaries.
  4. Update documentation, examples, and architecture decisions when behavior changes.
  5. Run relevant local validation and retain useful output for review.
Keep the increment reviewable: unrelated refactoring and broad formatting changes make requirement traceability and defect isolation harder.

Check

Goal: decide whether the implementation and its evidence satisfy the planned outcome.

CheckEvidenceDecision question
Requirement coverageLinked requirement, implementation, and acceptance testIs the intended behavior present and observable?
Implementation qualityCode review and automated review reportIs the solution correct, maintainable, and appropriately scoped?
Unit behaviorPassing focused testsAre decisions and edge cases covered?
Integration behaviorPassing boundary or workflow testsDo affected components still cooperate correctly?
Risk controlsSecurity, privacy, reliability, and operational checksAre relevant non-functional requirements satisfied?

Act

Goal: respond to the evidence and improve both the product and the next cycle.

When the check passes

  • Record the accepted version and approval evidence.
  • Release through the defined deployment path.
  • Confirm post-deployment health and expected behavior.
  • Move operational observations into the improvement backlog.

When the check does not pass

  • Translate findings into specific changes with owners.
  • Return to Do for implementation or to Plan if the requirement itself is unclear.
  • Repeat affected tests and reviews; do not reuse stale evidence.
Cycle completion: a release is not the end of PDCA. Production feedback, incidents, and measured outcomes become inputs to the next Plan phase.
