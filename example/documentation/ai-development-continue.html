AI DEVELOPMENT / Continue across chats

Use a repository handoff, not chat history

A new chat does not automatically have the reliable working state of an old chat, and different assistants do not share conversation memory. Keep a short CONTINUE.md at the repository root as the reviewed handoff between people, chats, IDEs, and assistants.

CONTINUE.md is project state, not permanent policy. Put durable commands and conventions in AGENTS.md; put the current goal, verified progress, blockers, and next actions in CONTINUE.md.

Commit the handoff when it should travel with the branch. Keep it untracked only when it contains personal scratch notes. Never put credentials, tokens, personal data, production records, or confidential logs in it.

Recommended CONTINUE.md template

# Continuation Last updated: YYYY-MM-DD HH:MM UTC Branch or worktree: feature/example ## Goal One observable outcome and its acceptance criteria. ## Current status - Completed: ... - In progress: ... - Not started: ... ## Decisions - Decision: ... Reason: ... ## Changed files - `path/to/file`: why it changed ## Verification - PASS — `exact command` - FAIL — `exact command` Error: concise, relevant error - NOT RUN — `command` Reason: ... ## Blockers and risks - ... ## Next actions 1. The smallest concrete next action. 2. The validation that follows it. ## Do not do - Scope boundaries and rejected approaches.

Use explicit states such as PASS, FAIL, and NOT RUN. “Tests look good” is not reproducible evidence. Remove obsolete statements instead of appending an endless diary.

Shared configuration for multiple tools

Keep the detailed rule once in AGENTS.md, then add the smallest adapter each tool understands.

Codex: AGENTS.md

# Working agreement - At the start of a task, read `CONTINUE.md` when it exists. - Verify its branch, file, and test claims against the current worktree. - Continue from the first incomplete action; do not redo completed work. - Before ending substantive work, update `CONTINUE.md` with exact commands, results, blockers, changed files, and the next action. - Never record secrets or sensitive data in the handoff.

GitHub Copilot: .github/copilot-instructions.md

Before editing, read `AGENTS.md` and `CONTINUE.md`. Treat `CONTINUE.md` as a fallible handoff: confirm its claims with Git status, the current files, and the listed verification commands. Update it after substantive work with exact results and next actions.

Any other assistant

Start the chat with an explicit prompt when the tool does not discover repository instructions automatically:

Read AGENTS.md and CONTINUE.md first. Verify the handoff against this worktree. Summarize the goal, current status, first incomplete action, and risks. Do not edit until the summary is correct.

End and resume a session

Before ending

  1. Run git status and inspect the diff.
  2. Record exact commands and outcomes, including skipped checks.
  3. State the first next action so the new chat does not need to infer it.
  4. Remove stale paths, decisions, and completed temporary steps.

At the start of the next chat

  1. Read the repository instructions and handoff.
  2. Compare the named branch, files, and status with the current worktree.
  3. Re-run cheap, drift-prone checks rather than trusting old output.
  4. Ask for clarification if the handoff conflicts with code or requirements.
Do not let an assistant certify its own history. The next session should verify important claims independently, especially security, migrations, external state, and tests that may depend on services.
