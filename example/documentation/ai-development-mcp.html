AI DEVELOPMENT / What MCP means

Model Context Protocol

MCP is an open standard for connecting an AI application to external data and actions. It lets an assistant discover capabilities through a consistent protocol instead of every assistant needing a custom integration for every service.

PartMeaningExample
HostThe AI application the user operates.Codex, Claude Code, or VS Code chat.
ClientThe connection maintained by the host for one server.The MCP connection and negotiated session inside the host.
ServerA program or service exposing selected capabilities.A documentation search service or issue tracker gateway.

MCP does not provide a model and is not itself a database. It is the connection contract between the AI host and a capability provider.

What a server can expose

PrimitivePurposeRisk example
ToolsActions the model may request, such as searching issues or creating a ticket.A write tool can change external state.
ResourcesReadable context such as a schema, document, or issue.Returned data may be confidential or contain malicious instructions.
PromptsReusable workflow templates supplied by the server.A template may conflict with project policy.

A local stdio server is started as a process by the host. A remote Streamable HTTP server is reached at a URL and commonly uses OAuth or bearer-token authentication.

Same server, different clients

These fictional examples connect a read-only documentation server at https://mcp.example.invalid/mcp. Replace it with an approved real service. Store the token in the environment—never in a committed configuration.

Codex: .codex/config.toml

[mcp_servers.project_docs] url = "https://mcp.example.invalid/mcp" bearer_token_env_var = "PROJECT_DOCS_TOKEN" enabled_tools = ["search", "read"] default_tools_approval_mode = "prompt"

The Codex desktop app, CLI, and IDE extension share Codex MCP configuration. Project-scoped configuration applies only in a trusted project.

VS Code and Copilot: .vscode/mcp.json

{ "servers": { "project-docs": { "type": "http", "url": "https://mcp.example.invalid/mcp", "headers": { "Authorization": "Bearer ${input:projectDocsToken}" } } }, "inputs": [ { "id": "projectDocsToken", "type": "promptString", "description": "Project documentation token", "password": true } ] }

Claude Code: .mcp.json

{ "mcpServers": { "project-docs": { "type": "http", "url": "https://mcp.example.invalid/mcp", "headers": { "Authorization": "Bearer ${PROJECT_DOCS_TOKEN}" } } } }

Claude prompts before first use of a project-scoped MCP configuration. Personal or experimental servers should use local/user scope rather than a committed file.

Security checklist

  • Trust the server operator, package publisher, executable, and endpoint—not just the displayed server name.
  • Grant the minimum tools and OAuth scopes. Prefer read-only capabilities until a write action is necessary.
  • Keep secrets in an approved secret store or environment variable and out of Git.
  • Review every write action, including its resolved target and parameters.
  • Treat resource and tool output as untrusted input that may contain prompt injection.
  • Check whether data leaves the workstation and whether the server retains requests.
  • Remove unused servers and revoke their tokens.
Local does not mean harmless. A stdio MCP server is executable code running with the permissions the host gives it. Review its command, package, version, filesystem access, and network access before starting it.
