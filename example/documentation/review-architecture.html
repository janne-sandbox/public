Code Review System — Architecture

System Architecture

The review system consists of a master orchestrator, a shared base agent, six language-specific agents, two optional Deepseek provider paths, requirement inputs, and Markdown report output.

Analysis Method: Hybrid Regex + Optional Deepseek

The review system supports two analysis approaches:

1. Regex-Based Analysis (Default, Always Available)

  • Method: Fast pattern matching using JavaScript regular expressions
  • Pros: Instant, deterministic, no dependencies, predictable
  • Cons: No semantic understanding, limited to exact patterns
  • Use cases: File counting, binary pattern detection (localStorage, SQL string concat)
  • Performance: 3-10 seconds per language

2. Deepseek LLM Analysis (Optional, Offline or Online)

INTERNET selects one of two explicit provider paths:

  • Offline: INTERNET=false DEEPSEEK=true uses local Ollama and verifies that DEEPSEEK_OLLAMA_MODEL is installed.
  • Online: INTERNET=true DEEPSEEK=true uses https://api.deepseek.com/chat/completions and requires DEEPSEEK_API_KEY.
  • Fallback: Missing credentials, models, or providers produce an explicit report status and retain deterministic checks.
  • C++/C# semantics: Up to six bounded code samples are reviewed; at most ten validated JSON findings are accepted.
Offline setup:
ollama pull deepseek-coder:1.3b-base ollama serve # Starts on localhost:11434
Online setup:
export DEEPSEEK_API_KEY='your-key' LANGUAGE=cpp INTERNET=true DEEPSEEK=true ./run-review.sh

Core Components

1. Master Orchestrator: ./run-review.sh

  • Entry point for all reviews
  • Parses parameters: LANGUAGE, INTERNET, DEEPSEEK
  • Creates a per-run output directory: ./review/results/<timestamp>/ or ./review/results/<timestamp>-N/ if the same minute is already in use
  • Spawns agent processes based on LANGUAGE parameter
  • Collects agent exit codes and reports results
  • Execution: ~10-20 seconds overhead for orchestration

2. Base Agent: ./review/agents/base-agent.js

Shared base class with common functionality:

  • loadRequirements() — loads .md files from ./requirements/
  • scanProjectFiles() — recursively scans for language-specific files
  • initializeDeepseek() — selects offline Ollama or the hosted API
  • reviewCodeWithDeepseek() — performs bounded semantic review for C++ and C#
  • extractRequirementCheckpoints() — parses requirement text for validation points
  • evaluateAgainstRequirement() — matches code against requirements
  • generateVerdict() — computes APPROVED|APPROVED WITH CHANGES|DENIED based on findings
  • writeReport() — formats Markdown output

3. Language Agents

Each extends the base class and adds language-specific checks:

4. Requirement Files

Agents load requirements from two locations:

  • ./requirements/functional/*.md — optional functional requirements
  • ./requirements/non-functional/*.md — GDPR, OWASP, WCAG, EN 50518 reqs

5. Finding Classification

Issues are classified by severity:

Finding { severity: 'critical' | 'major' | 'minor', title: string, file: string, line: number | 'N/A', description: string, requirement: string // which REQ/REVIEW applies } Change Suggestion { title: string, file: string, language: string, // ts, java, rs, tsx, etc. reason: string, codeSnippet: string }

Agent Execution Flow (per language)

  1. Load requirements — read all .md files from ./requirements/
  2. Scan project — recursively find all files matching language pattern (e.g., *.ts)
  3. Extract checkpoints — parse requirement text for validation points
  4. Initialize provider — select local Ollama or hosted Deepseek when requested
  5. Evaluate code — run deterministic checks and optional semantic C++/C# review
  6. Classify issues — categorize findings by severity
  7. Suggest changes — provide code fixes for identified issues
  8. Generate verdict — compute final APPROVED|APPROVED WITH CHANGES|DENIED
  9. Write report — output Markdown file to the current run folder under ./review/results/<timestamp>/
Performance note: Agents sample code (first 10-15 files) for pattern detection to keep execution fast. Critical checks are exhaustive; best-practice checks are sampled.

Requirements-Driven Design

The review system is explicitly requirements-driven. Every issue raised is traced back to a specific requirement (functional, non-functional, or review-specific). This ensures:

  • Traceability: Every finding references the requirement it violates
  • Consistency: All reviews check the same set of requirements
  • Clarity: Developers know exactly what standard they should meet
  • Extensibility: Adding new requirements automatically adds new checks

Example: REQ-006 (TypeScript Compile-Time Safety) contains a checkpoint about strict: true in tsconfig.json. The TypeScript agent checks for this and raises a major finding if missing, referencing REQ-006.
