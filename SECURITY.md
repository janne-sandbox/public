# Security Policy

## Supported Versions

Security updates are provided for the following versions:

| Version | Supported |
|---|---|
| Latest release | Yes |
| Previous releases | No |

Users should upgrade to the latest available version before reporting an issue.

## Reporting a Security Vulnerability

Do not report security vulnerabilities through public GitHub issues, pull requests, or discussions.

Report vulnerabilities privately using one of the following methods:

- GitHub private vulnerability reporting under **Security → Report a vulnerability**
- Email: `security@example.com`

Please include as much of the following information as possible:

- A clear description of the vulnerability
- The affected component and version
- Steps required to reproduce the issue
- A proof of concept, if available
- The potential impact
- Suggested mitigations or fixes
- Relevant logs, screenshots, or configuration details

Do not include sensitive personal, customer, patient, or production data in the report.

## What to Expect

After receiving a report, the maintainers will aim to:

1. Acknowledge the report within five business days.
2. Investigate and validate the reported issue.
3. Assess its severity and potential impact.
4. Develop and test a fix where appropriate.
5. Notify affected users and publish a security advisory when necessary.

The time required to resolve an issue depends on its severity and complexity.

## Responsible Disclosure

Please allow the maintainers reasonable time to investigate and resolve the issue before publicly disclosing it.

Security researchers must not:

- Access, modify, or delete data belonging to other users
- Test against production systems without explicit permission
- Perform denial-of-service or load-testing attacks
- Use automated tools that generate excessive traffic
- Attempt social engineering, phishing, or physical attacks
- Publicly disclose an unresolved vulnerability

## Scope

This policy applies only to code and components maintained in this repository.

Third-party libraries, services, infrastructure, and dependencies may be governed by their own security policies.

## Safe Harbor

Security research conducted in good faith and in accordance with this policy will be considered authorized.

The maintainers will not pursue legal action against researchers who:

- Make a reasonable effort to avoid privacy violations and service disruption
- Report findings promptly and privately
- Do not exploit vulnerabilities beyond what is necessary to demonstrate them
- Comply with applicable laws and this security policy

## Security Updates

Security fixes may be released through:

- New software releases
- GitHub security advisories
- Dependency updates
- Repository announcements or release notes
