Code Review System — Java Agent

Java Agent Architecture

The Java Agent extends the base ReviewAgent class and implements language-specific checks for Spring Boot 3.2 / Java 21 projects. It focuses on modern Java patterns, security compliance (OWASP), and proper use of frameworks.

Key Responsibilities

  • Look for modern Java patterns such as records and text blocks
  • Check Jackson usage for JSON serialization (no manual JSONObject construction)
  • Detect SQL injection vulnerabilities (string concatenation in queries)
  • Verify parameterized queries and prepared statements
  • Check test coverage (target ≥30% of classes)
  • Compare Java code with relevant functional requirements

Class Structure

class JavaAgent extends ReviewAgent { constructor(internetEnabled = false) async run(projectRoot, requirementsDir, outputDir, timestamp) _evaluateJavaCoverage(javaFiles) _checkModernJavaPatterns(javaFiles) _checkJacksonUsage(javaFiles) _checkOWASPCompliance(javaFiles) _checkTestCoverage(javaFiles) _checkFunctionalRequirements(javaFiles) }

Check Methods

1. Modern Java Patterns Check

_checkModernJavaPatterns() — Verifies Java 21 language features

  • Features checked: Records, text blocks, switch expressions
  • Severity: MINOR
  • Requirement: REQ-033 Java 21 Adoption
  • Records: Immutable value types for data classes (replace @Data)
  • Text blocks: Multi-line strings (cleaner than concatenation)
  • Switch expressions: Simpler, more expressive than traditional switch statements

2. Jackson Usage Check

_checkJacksonUsage() — Ensures proper JSON handling

  • Severity: MAJOR
  • Requirement: REQ-035 Jackson Framework Usage
  • What it detects: Manual JSONObject construction, string-based JSON building
  • Finding if failed: "Manual JSON construction detected" → suggests using Jackson ObjectMapper with record classes

3. SQL Injection Prevention Check

_checkOWASPCompliance() includes SQL injection detection

  • Severity: CRITICAL (security vulnerability)
  • Requirement: OWASP-REQ-001 Injection Prevention
  • What it detects: String concatenation in SQL queries (e.g., "SELECT * FROM users WHERE id = " + id)
  • Finding if found: "Potential SQL injection vulnerability" → blocks APPROVED verdict
  • Solution: Use prepared statements or JPA queries with named parameters

4. Test Coverage Check

_checkTestCoverage() — Verifies minimum test file ratio

  • Severity: MINOR (advisory heuristic below 30%)
  • Requirement: REQ-058 Test Coverage ≥30%
  • How it works: Counts *.java and *Test.java files, calculates ratio
  • Target: At least 30% of classes have corresponding test classes

5. Functional Requirements Check

_checkFunctionalRequirements() — Matches code against project requirements

  • Severity: MINOR
  • How it works: Loads requirements from ./requirements/functional/, checks if Java files address them
  • Coverage report: Lists which requirements are reflected in code

Implementation Method

Technology Stack: Node.js 20 + regex-based pattern matching + optional Deepseek LLM

  • Default (Regex): Fast pattern matching (instant, no dependencies)
  • Enhanced (Deepseek): Offline LLM analysis via Ollama (if available on localhost:11434)
  • Engine: Regex for speed + Deepseek for semantic understanding (hybrid approach)
  • No internet required: Deepseek runs locally via Ollama, no cloud API calls

Analysis Method

Default Mode (Always Available): Regex-based pattern matching

  • Scans for: SQL string concatenation, Jackson usage, records, text blocks, switch expressions
  • Performance: ~5-10 seconds
  • Does NOT use: javac, checkstyle, spotbugs, or any Java tools

Enhanced Mode (If Deepseek Available): Hybrid regex + LLM analysis

  • Uses Deepseek to filter language-appropriate requirements (skip React/TypeScript requirements)
  • Uses the local model to select requirements relevant to Java
  • Performance: ~30-45 seconds (due to LLM inference)
  • The Java pattern checks remain deterministic and regex-based
  • Requires: Ollama running locally with deepseek-coder model

What This Means

The Java Agent:

  • ✓ Uses regex patterns for fast, deterministic code analysis
  • ✓ Can detect string concatenation patterns and simple code structures
  • ✗ Does NOT use Java compiler (no bytecode analysis, no type inference)
  • ✗ Does NOT use AST parser (no semantic understanding of code)
  • ✗ Does NOT use static analysis tools like checkstyle, spotbugs, or sonarqube
  • ✗ Cannot understand method signatures, inheritance chains, or type resolution

File Scanning Pattern

The agent scans for Java files matching **/*.java. The tree below is illustrative; these names are not taken from a reviewed project.

Critical Issues: SQL Injection Example

Critical finding (blocks APPROVED): SQL injection detected

❌ Bad code (detected as CRITICAL):

@GetMapping("/orders/{id}") public ResponseEntity<Order> getOrder(@PathVariable String id) { String query = "SELECT * FROM orders WHERE id = " + id; // Critical return repository.findByCustomQuery(query); }

✓ Good code (passes check):

@GetMapping("/orders/{id}") public ResponseEntity<Order> getOrder(@PathVariable String id) { return ResponseEntity.ok(orderRepository.findById(id)); // Parameterized by the repository }

Performance Characteristics

  • Execution time: 5-10 seconds
  • Sample size: First 15 files for pattern detection, all files for counting
  • Memory usage: ~20-50 MB (typical)
  • Network access: None required (offline mode)

Limitations & Future Improvements

  • Current: Regex-based pattern matching (fast, not AST-based)
  • Could improve: Use JavaParser library for AST analysis
  • Current: No support for Gradle projects
  • Could improve: Parse build.gradle for dependency analysis
  • Current: No CVE checking for Maven dependencies
  • Could improve: Integrate OWASP Dependency Check when INTERNET=true
  • Current: Samples first 15 files for performance
  • Could improve: Full AST traversal for critical checks like SQL injection
